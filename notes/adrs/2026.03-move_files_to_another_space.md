# ADR: Move files to another space

## Context

SimpleDMS currently supports moving files inside the current space, but not across spaces.

Product requirements for this ADR:

- It must be possible to move a single file from Inbox and from Files view.
- It must be possible to move multiple files at once.
- There must be two target modes:
  - move to Inbox of the target space,
  - move directly to processed files in the target space.
- There must be an option to preserve metadata.
- There must be an option to auto-create metadata in the target space when missing.

Relevant technical constraints in current model:

- Files are space-scoped (`file.space_id`) and cannot belong to multiple spaces at the same time.
- Metadata is also space-scoped:
  - document types,
  - tags,
  - fields (properties).
- Existing move logic is folder-mode oriented and mostly same-space.

## Decision

### Operation scope and entry points

Introduce a new cross-space command `Move files to another space` with:

- single-file entry points:
  - Inbox file actions,
  - Files view file actions;
- multi-file entry points:
  - Inbox selection toolbar,
  - Files view selection toolbar.

v1 scope:

- only regular files can be moved across spaces,
- directories are excluded from cross-space move in this ADR,
- source and target space must be different.

### Target modes

The command supports two explicit modes:

1. `target_inbox`

- Move file(s) to the target space and mark them as Inbox files (`is_in_inbox=true`).
- Parent directory is the target space root directory.

2. `target_processed`

- Move file(s) to the target space and mark them as processed (`is_in_inbox=false`).
- Use a required folder picker to select destination directory in target space.
- Root directory is available as a picker option.
- In non-folder mode, picker resolves to target root directory only.

### Metadata options

Add two options to the move command:

- `preserve_metadata` (default: `false`)
- `auto_create_missing_metadata` (default: `false`, only effective when `preserve_metadata=true`)

Metadata categories and behavior:

1. Document type

- Preserve by name match in target space.
- If no match exists and `auto_create_missing_metadata=true`, create document type in target space.
- If no match exists and `auto_create_missing_metadata=false`, clear document type on moved file.

2. Tags

- Preserve by semantic identity using tag path (`group path + tag name`) inside target space and reuse
  existing tags.
- If `auto_create_missing_metadata=true`, create missing tag groups/tags in target space and assign them.
- If `auto_create_missing_metadata=false`, keep only matched tags and drop unmatched tags.

3. Fields (properties)

- Preserve by identity key (`name + type + unit`) in target space and reuse existing fields.
- If `auto_create_missing_metadata=true`, create missing fields and assign copied values.
- If `auto_create_missing_metadata=false`, keep only matched fields and drop unmatched fields.

If `preserve_metadata=false`, clear all file metadata during move:

- document type,
- tag assignments,
- field assignments/values.

### Processing and transaction semantics

- Single-file move is atomic in one transaction.
- Bulk move is processed per file with one transaction per file.
- No partial updates within one file move.
- Bulk operations may partially succeed across files; response returns per-file results, summary counts, and
  a clear error message per failed file.

### Authorization and safety

For each moved file, enforce:

- permission to access and move from source space,
- permission to create/update files in target space,
- file still exists and is not deleted,
- destination naming/conflict rules of target space.

### UX behavior

Move dialog includes:

- target space,
- target mode (`Inbox` or `Processed files`),
- required destination folder picker (for processed mode),
- `Preserve metadata` checkbox,
- `Auto-create missing metadata` checkbox (enabled only when preserve is enabled).

Result feedback includes:

- number of moved files,
- number of skipped/failed files,
- metadata migration summary (matched, auto-created, dropped).

## Rationale

- Two explicit target modes cover both re-triage workflows (Inbox) and direct final placement workflows
  (Processed files).
- Per-file transactions in bulk mode prevent one bad file from blocking a large user operation.
- Separate metadata options keep behavior predictable:
  - strict move without metadata,
  - best-effort preserve,
  - preserve with controlled schema evolution in target space.
- Reusing existing target metadata first and only creating missing entries avoids unnecessary duplication.

## Consequences

Positive:

- users can rebalance and reorganize work between spaces from both Inbox and Files views,
- bulk move reduces repetitive manual actions,
- metadata behavior is explicit and user-controlled.

Trade-offs:

- metadata mapping logic adds complexity,
- bulk operations can result in mixed outcomes and need clear reporting,
- auto-creating document types/tags/fields can increase taxonomy size in target spaces.

## Alternatives considered

1. Always drop metadata on cross-space move
   Rejected: does not satisfy requirement to preserve metadata.

2. Always preserve and always auto-create metadata
   Rejected: too implicit, can unexpectedly pollute target-space metadata model.

3. All-or-nothing transaction for bulk move
   Rejected: poor UX for larger selections where one problematic file blocks all others.

4. Auto-create only tags/fields, not document types
   Rejected: inconsistent metadata migration behavior for users.

## Acceptance criteria

- A single file can be moved from Inbox to a different space.
- A single file can be moved from Files view to a different space.
- Multiple files can be selected and moved in one operation from Inbox.
- Multiple files can be selected and moved in one operation from Files view.
- Mode `target_inbox` moves file(s) to target space with `is_in_inbox=true`.
- Mode `target_processed` moves file(s) to target space with `is_in_inbox=false`.
- Mode `target_processed` requires folder selection in target space via folder picker.
- `preserve_metadata=false` clears document type, tags, and fields on moved files.
- `preserve_metadata=true` preserves document type/tags/fields when matching metadata exists in target
  space.
- `auto_create_missing_metadata=true` creates missing document types, tags, and fields in target space and
  preserves them on moved files.
- `auto_create_missing_metadata=false` does not create metadata and drops unmatched assignments.
- For bulk moves, each file is atomic; operation returns success/failure summary, per-file failures, and
  clear per-file error messages.

## Test coverage required

- Single-file move from Inbox to another space in both target modes.
- Single-file move from Files view to another space in both target modes.
- Bulk move from Inbox with mixed valid/invalid files.
- Bulk move from Files view with mixed valid/invalid files.
- Metadata preserve off: metadata is cleared.
- Metadata preserve on + auto-create off: only matched metadata is retained.
- Metadata preserve on + auto-create on: missing document types/tags/fields are created and assigned.
- Document type missing in target with auto-create off: document type is cleared and operation still
  succeeds.
- Permission denial on source or target space blocks that file and reports clear error.
- Naming conflicts in target follow existing conflict rules and report per-file failure.
- Processed mode requires destination folder selection and rejects missing/invalid folder.

- [ ] Add cross-space move command input model with:
  - [ ] file IDs (single and bulk),
  - [ ] target space ID,
  - [ ] target mode,
  - [ ] target directory ID (required for `target_processed`, root allowed),
  - [ ] metadata options.

### 2) UI integration

- [ ] Add single-file action in Inbox file actions.
- [ ] Add single-file action in Files view file actions.
- [ ] Add multi-selection state/actions for Inbox list.
- [ ] Add multi-selection state/actions for Files list.
- [ ] Add move dialog with mode and metadata options.

### 3) Domain service

- [ ] Implement `MoveFilesToSpace` domain service.
- [ ] Validate permissions and source/target invariants per file.
- [ ] Execute per-file transactional move.
- [ ] Return structured per-file result and summary.

### 4) Metadata mapping and creation

- [ ] Add tag mapper by semantic path.
- [ ] Add field mapper by `name + type + unit`.
- [ ] Add document-type mapper by name.
- [ ] Add optional creation for missing document types/tags/fields.

### 5) Observability and events

- [ ] Add structured logs for move mode/options and outcomes.
- [ ] Emit domain/UI event for list refresh in source and target contexts.

### 6) Tests

- [ ] Add integration tests for single and bulk flows.
- [ ] Add metadata mapping/creation tests.
