# ADR: Passkey support as a login method

## Context

SimpleDMS currently supports sign-in with email and password.

Passkeys are intended to be the primary phishing-resistant login method for accounts that enable them.

Product and rollout requirements:

- Passkeys are available by default as an optional feature.
- The existing login form (email + password) stays available for users without passkeys.
- A separate `Sign in with passkey` button is added for username-less passkey sign-in.
- Passkey enrollment is supported from authenticated account settings only (not during sign-up).
- If passkey login is enabled for an account, password login is no longer allowed for that account.
- Recovery uses recovery codes and admin-assisted recovery.
- Current "remember me" behavior remains unchanged: valid session means no re-login.
- Tenant-level policy must be represented in the ADR.
- WebAuthn host/domain configuration must come from environment configuration and also be usable for
  transactional email links.

Deployment context:

- SaaS currently serves `app.simpledms.ch`, `app.simpledms.eu`, and `simpledms.app` from one runtime.
- In the future, `app.simpledms.ch` and `app.simpledms.eu` may be hosted separately.

## Decision

### Authentication model

Use a hybrid login UI:

- Keep email + password form for users who do not use passkeys.
- Add a dedicated `Sign in with passkey` action that starts username-less WebAuthn authentication.

Username-less passkey sign-in is implemented with discoverable credentials (resident credentials), so the
user can select a passkey directly in the authenticator UI without entering an email first.

### Account-level behavior

Introduce account-level passkey state and enforce:

- If account has passkey login enabled, password sign-in is rejected for that account.
- If account does not have passkey login enabled, existing password login continues to work.

### Enrollment scope

- Passkey registration and management are available only in authenticated account settings.
- Sign-up flow is unchanged for v1.

### Recovery and lockout handling

When passkey login is enabled for an account:

- Password reset is not used as a fallback login path.
- Recovery is done through one-time recovery codes.
- If recovery codes are unavailable, admin-assisted recovery is used.

### Tenant policy semantics

Add tenant-level passkey policy and compute effective account requirements with a strictest-wins rule:

- If any active tenant membership of an account requires passkey login, the account must use passkey login.
- This rule applies across all active tenant memberships, not only default tenant.

This avoids inconsistent and weaker behavior for accounts belonging to multiple organizations.

### Tenant-enforced bootstrap for not-yet-enrolled accounts

When tenant policy requires passkeys but an account has not enrolled any passkey yet:

- Password sign-in is allowed only as a bootstrap path to complete passkey setup.
- The resulting session is temporary and restricted to passkey setup/recovery actions and sign-out.
- After the first passkey is registered, password sign-in is rejected for that account.

This avoids lockout after policy changes while still converging quickly to passkey-only access.

### WebAuthn security defaults

Use the following WebAuthn defaults:

- `userVerification = required`
- `attestation = none`
- discoverable/resident credentials preferred

This provides strong user verification, avoids attestation/privacy complexity, and enables username-less
sign-in UX.

### Domain and RP strategy

Use canonical deployment origin from environment configuration and enforce canonical host redirects.

Configuration:

- `SIMPLEDMS_PUBLIC_ORIGIN` (required): canonical public origin for this deployment.
- `SIMPLEDMS_WEBAUTHN_RP_ID` (optional): explicit RP ID override; default derives from
  `SIMPLEDMS_PUBLIC_ORIGIN` host.
- `SIMPLEDMS_WEBAUTHN_RP_NAME` (optional): display name for authenticator prompts.

Operational strategy:

- Treat `app.simpledms.eu` and `app.simpledms.ch` as separate WebAuthn realms (region-pinned RP IDs).
- Use `simpledms.app` as discovery/redirect entry point, not as a passkey realm of record.
- Accounts are bound to one home region/auth domain.
- Moving an account between regions requires passkey re-enrollment.

## Rationale

- A dedicated passkey button provides a modern, low-friction path while preserving compatibility for existing
  password users.
- Username-less passkey flow removes email entry from the critical path and aligns with platform passkey UX.
- Disabling password login for passkey-enabled accounts enforces a clear security posture and avoids fallback
  weakening.
- Recovery codes plus admin-assisted recovery provide a practical lockout safety net without reintroducing
  password fallback.
- Strictest-wins tenant policy is safer for multi-tenant users and prevents policy bypass via tenant context
  selection.
- Region-pinned RP IDs align with future EU/CH infrastructure split and WebAuthn domain boundaries.
- Canonical public origin also supports consistent transactional link generation.

## Consequences

Positive:

- phishing-resistant login path available by default,
- clear migration path from password-first to passkey-first accounts,
- future-ready regional hosting model for passkeys.

Trade-offs:

- passkeys are not portable across different RP IDs/TLDs,
- account moves across regions require explicit credential re-enrollment,
- admin-assisted recovery requires operational procedures and authorization controls,
- strictest-wins policy can force passkey adoption for users in mixed-policy tenants.
- bootstrap sessions add routing complexity and need explicit endpoint allowlisting.

## Alternatives considered

1. Email-first passkey login only
   Rejected: keeps extra friction and does not exploit discoverable credential UX.

2. Keep password fallback for passkey-enabled accounts
   Rejected: weakens the security objective for passkey-enabled mode.

3. Apply tenant policy only for default tenant
   Rejected: creates bypass and inconsistency for multi-tenant accounts.

4. Single global auth domain for all regions
   Rejected: conflicts with planned region separation and data residency direction.

## Acceptance criteria

- Login page contains:
  - existing email + password form,
  - `Sign in with passkey` action for username-less flow.
- Passkey registration is possible from authenticated account settings only.
- If passkey login is enabled for an account, password login for that account is rejected.
- If passkey login is not enabled for an account, password login remains available.
- Recovery for passkey-enabled accounts supports recovery codes and admin-assisted recovery.
- Effective account policy uses strictest-wins across active tenant memberships.
- For tenant-enforced accounts without enrolled passkeys, bootstrap password sign-in creates a temporary
  setup session restricted to passkey setup/recovery/sign-out.
- WebAuthn requests use `userVerification=required` and `attestation=none`.
- Canonical host/origin is configurable via environment and reused for transactional link generation.
- For SaaS multi-domain setup, passkeys are region-pinned; cross-region moves require re-enrollment.
- Existing session/remember-me behavior remains unchanged.

## Test coverage required

- Username-less passkey sign-in succeeds for account with passkey enabled.
- Password sign-in is rejected for account with passkey enabled.
- Password sign-in still succeeds for account without passkey enabled.
- Passkey enrollment is blocked when unauthenticated and allowed in authenticated settings.
- Recovery code flow works once per code and invalidates used recovery codes.
- Admin-assisted recovery flow is access-controlled and auditable.
- Effective policy computation for multi-tenant account follows strictest-wins semantics.
- Tenant-enforced account without passkeys can bootstrap via password sign-in only into restricted setup
  session.
- Restricted setup session blocks non-setup routes until passkey enrollment is completed.
- Canonical-host redirect prevents creating/authenticating credentials on non-canonical host.
- RP ID derivation from environment works; override behavior works.
- Existing persistent session behavior is unchanged (valid session bypasses login).

## Implementation checklist

### 1) Configuration and canonical origin

- [ ] Add env vars in `.env.sample`:
  - [ ] `SIMPLEDMS_PUBLIC_ORIGIN`
  - [ ] `SIMPLEDMS_WEBAUTHN_RP_ID` (optional)
  - [ ] `SIMPLEDMS_WEBAUTHN_RP_NAME` (optional)
- [ ] Parse these vars in server startup and expose them via runtime config model (similar to
      `AllowInsecureCookies`).
- [ ] Add canonical-host enforcement (redirect to `SIMPLEDMS_PUBLIC_ORIGIN` host) before auth flows so
      passkey registration/authentication always happens on the correct host.
- [ ] Reuse `SIMPLEDMS_PUBLIC_ORIGIN` when generating transactional email links.

### 2) Database and ent schema

- [ ] Add dedicated passkey credential entity in `db/entmain/schema` (new schema file), including at least:
  - [ ] `account_id`
  - [ ] credential ID (unique)
  - [ ] public key / credential material
  - [ ] sign counter
  - [ ] metadata (name/label, transports, last used at)
- [ ] Add account-level passkey login state (enabled/disabled, timestamps).
- [ ] Add tenant-level policy field for passkey requirement (default: optional/not required).
- [ ] Add challenge persistence for WebAuthn ceremony state (short TTL, single-use, type-aware).
- [ ] Reuse existing recovery-code storage or add dedicated passkey recovery-code fields for v1.
- [ ] Run `go generate ./...` to regenerate ent and model artifacts.

### 3) Domain model and policy computation

- [ ] Add passkey domain/service layer in `model/account` for:
  - [ ] begin registration
  - [ ] finish registration
  - [ ] begin authentication
  - [ ] finish authentication
  - [ ] credential revoke/list
- [ ] Add strictest-wins policy resolver:
  - [ ] inspect all active tenant memberships
  - [ ] if any tenant requires passkey, effective policy requires passkey.
- [ ] Integrate policy check into password auth path so password sign-in is rejected when account/effective
      policy requires passkey.
- [ ] Keep current remember-me/session behavior unchanged.

### 4) Auth actions and router wiring

- [ ] Add auth actions/commands in `action/auth` for WebAuthn begin/finish endpoints:
  - [ ] unauthenticated passkey sign-in begin
  - [ ] unauthenticated passkey sign-in finish
  - [ ] authenticated passkey registration begin
  - [ ] authenticated passkey registration finish
  - [ ] passkey delete/rename/list for settings UI
- [ ] Add recovery-code login flow for passkey-enabled accounts (separate from password flow).
- [ ] Add admin-assisted recovery action(s) with explicit authorization checks and auditing hooks.
- [ ] Register new actions in `action/auth/actions.go` and route registration.
- [ ] Extend unauthenticated allowlist in `server/router.go` for passkey sign-in and recovery endpoints.

### 5) UI and frontend ceremony support

- [ ] Update `action/auth/sign_in_page.go`:
  - [ ] keep existing email/password form
  - [ ] add `Sign in with passkey` button
  - [ ] add recovery-code entry path
- [ ] Add account settings UI to manage passkeys (create/list/delete/rename) and recovery codes.
- [ ] Add frontend WebAuthn helper script (base64url encode/decode + ceremony calls) and wire it into sign-in
      and settings views.
- [ ] Add UX messaging for unsupported browsers or disabled platform authenticators.

### 6) Security and operational hardening

- [ ] Enforce WebAuthn defaults in server options:
  - [ ] `userVerification=required`
  - [ ] `attestation=none`
  - [ ] discoverable credential preference enabled
- [ ] Ensure all challenges are short-lived, single-use, and bound to ceremony type.
- [ ] Store only required credential data; avoid storing unnecessary attestation payloads.
- [ ] Add rate limiting / anti-abuse checks for begin and finish endpoints.

### 7) Tests and rollout safety

- [ ] Add integration tests for passkey sign-in success/failure and password-blocking semantics.
- [ ] Add tests for strictest-wins tenant policy across multiple memberships.
- [ ] Add tests for canonical-host enforcement and RP ID configuration behavior.
- [ ] Add tests for recovery-code one-time usage and admin-assisted recovery authorization.
- [ ] Add migration/backward-compat tests so existing password users continue to sign in unchanged.

### 8) Documentation and migration notes

- [ ] Document operational setup (RP ID, canonical origin, region-pinned domains) in technical docs.
- [ ] Document account home-region binding and re-enrollment requirement after region move.
- [ ] Add support runbook for admin-assisted recovery verification and actions.

## Execution plan (single large PR)

The implementation is delivered as one PR, but developed in the sequence below to keep risk and rework low.

1. Runtime config and canonical origin plumbing

- Add and wire env vars (`SIMPLEDMS_PUBLIC_ORIGIN`, `SIMPLEDMS_WEBAUTHN_RP_ID`,
  `SIMPLEDMS_WEBAUTHN_RP_NAME`).
- Add canonical-host redirect logic and tests.
- Keep behavior no-op if passkey endpoints are not used yet.

2. Schema foundation and generated code

- Add ent schema for passkey credentials, ceremony challenges, account passkey mode, tenant passkey policy,
  and recovery artifacts.
- Run `go generate ./...` and ensure generated code compiles.
- Add migration/backward-compat tests for existing password users.

3. WebAuthn service layer

- Add model/service objects for begin/finish registration and begin/finish authentication.
- Implement challenge issuance/validation, sign counter handling, and credential lifecycle.
- Enforce defaults (`userVerification=required`, `attestation=none`, discoverable credentials preferred).

4. Policy and password-path enforcement

- Implement strictest-wins effective policy across active tenant memberships.
- Enforce "password login blocked when passkey-required/effective-passkey-enabled" in auth model.
- Keep existing session and remember-me semantics unchanged.

5. Action endpoints and router integration

- Add auth actions for passkey sign-in begin/finish.
- Add settings actions for passkey register/list/delete/rename.
- Add recovery-code and admin-assisted recovery actions.
- Register endpoints and update unauthenticated route allowlist where required.

6. UI wiring

- Update sign-in page to show existing form plus `Sign in with passkey` button.
- Add settings UI for passkey management and recovery codes.
- Add frontend ceremony helper (WebAuthn browser API, base64url conversions, error mapping).

7. Security hardening and audit behavior

- Add anti-abuse/rate-limit hooks for begin/finish auth ceremonies.
- Add audit-relevant events for credential create/use/delete and admin-assisted recovery actions.
- Verify sensitive data handling and avoid unnecessary attestation storage.

8. Full integration tests and stabilization

- Add tests for success/failure paths, policy interactions, canonical host constraints, and recovery code
  one-time use.
- Run full test suite and fix regressions.
- Final consistency pass on docs and i18n strings introduced by new UI/actions.

9. PR packaging for review

- Keep one PR, but structure commits by the phases above for easier review.
- PR description should map commits to ADR decisions and acceptance criteria.
- Include a short operator checklist for SaaS region/domain configuration at deploy time.
